# 为什么 toB AI 需要 FDE
{: .fs-6 }

{: .note }
本文讨论企业级（toB）场景下，**Forward Deployed Engineer（前沿部署工程师，FDE）** 在 AI 落地中的角色。可与本站首页对 [AI FDE](https://www.palantir.com/docs/foundry/ai-fde/overview) 的介绍对照阅读。

## toB AI 的典型困境

企业客户买的往往不是「一个模型」，而是一套能进生产环境、能审计、能协作的工作流。常见卡点包括：

- **需求在业务现场**：法务、运营、一线团队的真实诉求，很难用一份 PRD 在总部写全。
- **数据与权限复杂**：多租户、合规、存量系统接口，决定了 AI 不能只做演示型 Demo。
- **交付要可运维**：上线之后还有监控、迭代、培训与责任边界，纯项目制交付容易断层。

{: .important }
toB 的核心矛盾是：**技术迭代在实验室，价值发生在客户现场。**

## FDE 解决的是什么

FDE（前沿部署工程师）把工程能力部署在离客户最近的位置，在真实环境里完成「翻译」工作：

| 维度 | 没有 FDE 时 | 有 FDE 时 |
|------|-------------|-----------|
| 需求 | 产品猜、销售传话 | 在现场把话术变成可执行规格 |
| 集成 | 集成商与产研脱节 | 同一人理解业务语义与 API/本体 |
| 迭代 | 大版本、长周期 | 小步快跑，对齐平台与数据现实 |

在 Palantir Foundry 语境下，[AI FDE](https://www.palantir.com/docs/foundry/ai-fde/overview) 进一步把自然语言操作平台的能力产品化——但**仍需要懂业务与平台的人**在一线把关。

## 为什么 toB AI 尤其需要这类角色

1. **AI 输出不确定**：需要有人定义验收标准、兜底流程与人机协同，而不是「模型上线即结束」。
2. **行业差异大**：金融、制造、政务的合规与数据形态不同，模板化交付往往不够。
3. **客户买的是结果**：toB 合同绑定的是 SLA 与业务指标，FDE 型角色对齐的是**结果责任**，而非单次交付物。

{: .highlight }
一句话：**toB AI 的价值 = 模型能力 × 现场工程化 × 持续运营。** FDE 负责后两项。

## 小结

toB AI 不需要更多「只会调 API 的 Demo」；需要的是能在客户现场把 AI 嵌进流程、数据与组织的人。FDE 正是这种角色的现代命名——无论是否使用 Foundry，**前沿部署**的逻辑都成立。
